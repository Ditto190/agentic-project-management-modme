# Scope Definition Document — Example Engagement

## 1. Executive Summary

Example Client Ltd is undertaking a data platform modernisation programme to replace their legacy on-premises data warehouse with a cloud-native lakehouse architecture. This engagement covers the discovery and design phases, producing a validated scope baseline, SoW, and delivery approach for the full implementation programme.

## 2. Engagement Context

### 2.1 Client Background

Example Client Ltd is a mid-size financial services organisation with approximately 2,000 employees. Their current data infrastructure is over 10 years old and cannot support the real-time analytics required by their digital transformation roadmap.

### 2.2 Business Problem Statement

The current data warehouse is unable to support real-time data ingestion, has high maintenance costs, and cannot scale to meet projected data volumes over the next 3 years.

### 2.3 Strategic Objectives

| # | Objective | Success Metric |
|---|-----------|---------------|
| 1 | Replace legacy data warehouse with cloud lakehouse | New platform live by Q4 2025 |
| 2 | Enable real-time analytics | Sub-5-minute data latency for key dashboards |
| 3 | Reduce data infrastructure operating cost | 30% cost reduction vs current state |

## 3. Scope Definition

### 3.1 In Scope

| # | In-Scope Item | Notes |
|---|--------------|-------|
| 1 | Cloud lakehouse architecture design and implementation | AWS-based |
| 2 | Migration of 15 priority data domains | As agreed in kick-off |
| 3 | Data quality framework implementation | Using Great Expectations |
| 4 | Self-service analytics layer | Power BI |
| 5 | Team training and knowledge transfer | 3-day programme |

### 3.2 Out of Scope

| # | Out-of-Scope Item | Rationale |
|---|------------------|-----------|
| 1 | Legacy system decommission | Client BAU responsibility post-migration |
| 2 | Non-priority data domains (40 domains) | Deferred to Phase 2 |
| 3 | Real-time streaming ingestion | Phase 2 enhancement |
